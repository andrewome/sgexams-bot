# DM the affected user after mute/kick/ban

> Kick/ban's DM ordering is superseded by ADR-0005 (DM sent before the action, past a `lookup()` gate). Mute is unaffected and this document still applies to it as written.

After a mute, kick, or ban, the bot now DMs the affected user (an "Action notice" — see `CONTEXT.md`) with the reason, duration if any, and which moderator performed it. This includes `WarnCommand`'s warn-threshold auto-escalation to ban/mute, not just directly-invoked commands — from the affected user's perspective there's no difference, and the notice matters more here since there's no moderator visibly typing a command to ask. The moderator field shows the bot itself for auto-escalation, matching what the DB log already records (`botId` as `modId`).

`DiscordMemberPort` (ADR-0002) gets a new `dm(userId, options: { embeds: EmbedBuilder[] })` method, returning the same `MemberActionResult` as its other methods (resolving the user to DM them naturally produces their tag, same as `ban`/`kick`/`timeout`). The port stays domain-ignorant - it doesn't know what a "mute notice" is, only how to send a DM. The actual embed content is built by a new `ModUtils.buildActionNoticeEmbed(...)`, not on `Command` itself: `Command` is the generic base class every command family uses (Starboard, MsgChecker, Birthday, not just moderation), so putting mute/kick/ban-specific content there would leak domain knowledge into an unrelated shared class. `ModUtils` already holds moderation-specific shared formatting (`formatDuration`), so this is the same role.

## Consequences

The DM is strictly best-effort and never blocks the underlying mute/kick/ban: a user with DMs disabled, or who has blocked the bot, or who has left the server, is common and expected, not an error condition. The moderator's confirmation embed gets a `Notified` field only when the DM fails (not on success, to avoid a redundant line in the common case), naming the DM mechanism explicitly and giving a friendly, generic reason ("may have DMs disabled or have blocked the bot") rather than the raw `DiscordAPIError` text - the actual error is logged instead (`DiscordMemberAdapter.toFailure` already does this for every port method, `dm()` included), keeping the embed readable rather than surfacing API internals to the moderator.

`CommandArgs` gains a `serverName` field, sourced from the guild name `OnMessageEventHandler` already reads but previously didn't pass through - needed so the DM can tell the user which server the action happened in (the bot may share many servers with them).

The moderator's identity in the DM is a `<@modId>` mention rather than a resolved tag - `CommandArgs` only carries the moderator's `userId` today, and a mention renders correctly in a DM without needing to add a tag field or an extra fetch.

All four sites (`MuteCommand`, `KickCommand`, `BanCommand`, `WarnCommand`'s two escalation branches) send the notice **only after** the action is confirmed to have succeeded, never before. An earlier version of this change sent kick/ban's notice *before* the removal, reasoning that a kicked/banned member would no longer be fetchable via `GuildMemberManager` afterward - true, but it meant a kick/ban that failed for any reason *other* than an invalid user (e.g. the bot's role sitting below the target's in the hierarchy) would still tell the user "you were banned" when nothing had actually happened. The real fix was narrower than reordering: `dm()` fetches through the client's *global* user manager (`client.users.fetch`), not the guild-scoped `GuildMemberManager` - DMing never depended on current guild membership in the first place, only the earlier implementation's choice of fetch method did. With that fixed, every command can safely and uniformly notify only after a confirmed success, which is also the more obviously correct behavior to read.
